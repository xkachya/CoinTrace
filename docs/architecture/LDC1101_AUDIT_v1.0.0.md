## 2. Аудит конфігурації реєстрів

### 2.1 Порівняння з MikroE SDK

| Регістр | Адреса | Arch doc v1.0.0 | MikroE SDK `ldc1101_default_cfg()` | Статус |
|---|---|---|---|---|
| RP_SET  | `0x01` | 0x07 (POR default) | **0x26** (RP_MAX=24kΩ/MIN=1.5kΩ) | ✅ Виправлено в хвилі 3 (ADR-LDC-001) |
| TC1     | `0x02` | — (POR default = 0x00) | **0x1F** (C1=0.75pF, R1=21.1kΩ, τ₁=15.8ns) | ✅ Додано в хвилі 3 |
| TC2     | `0x03` | — (POR default = 0x00) | **0x3F** (C2=3pF, R2=30.5kΩ, τ₂=91.5ns) | ✅ Додано в хвилі 3 |
| DIG_CFG | `0x04` | depends on resp_time_bits | **0xD4** (MIN_FREQ=0xD0 + RESP_TIME=0x04=768) | ⚠️ MIN_FREQ ✅, RESP_TIME 6144≠768 (LA-4) |
| ALT_CONFIG | `0x05` | 0x00 writes | 0x00 | ✅ |
| D_CONFIG   | `0x0C` | 0x00 writes | 0x00 | ✅ |

**Висновок §2.1:** Після хвилі 3 регістрова конфігурація архітектурного документа синхронізована з MikroE SDK за RP_SET, TC1, TC2 та MIN_FREQ. Єдина задокументована розбіжність — RESP_TIME (LA-4).

### 2.2 RESP_TIME: архітектура vs MikroE SDK

```
Архітектура (ldc1101.json default):   resp_time_bits = 7  → RESP_TIME = 6144 cycles
MikroE SDK ldc1101_default_cfg():     DIG_CFG bit[2:0]=4 → RESP_TIME = 768 cycles

ConversionTime при fSENSOR = 300 kHz:
  Архітектура:  6144 / (3 × 300 000) = 6.84 мс
  MikroE SDK:    768 / (3 × 300 000) = 0.85 мс

SNR відношення (більший RESP_TIME = більше усереднення):
  SNR ∝ √RESP_TIME
  SNR(6144) / SNR(768) = √(6144/768) = √8 = 2.83×
  → Архітектурний default дає SNR на 2.83× краще ніж MikroE SDK default
  → Ціна: 8× довший час конверсії (6.84 мс vs 0.85 мс)

Обидва вкладаються в бюджет update() = 20 мс при 50 Hz:
  6.84 мс << 20 мс ✅
  0.85 мс << 20 мс ✅
```

**Висновок §2.2:** Вибір 6144 cycles є свідомим покращенням точності над MikroE default. **Повинен бути задокументований як ADR** (поточний стан: задокументовано в тексті §2 архітектурного doc але без окремого ADR-номера).

### 2.3 TC1/TC2 — фізичний сенс та верифікація

TC1 і TC2 — компенсаційні постійні часу інтегратора LDC1101. Вони "обрізають" паразитні ємності плати щоб отримати правильне C_EFF для LC-осцилятора:

```
τ₁ = R1 × C1 = 21.1 kΩ × 0.75 pF = 15.8 ns
τ₂ = R2 × C2 = 30.5 kΩ × 3 pF   = 91.5 ns

Фізичний сенс: LDC1101 використовує τ₁, τ₂ для корекції вхідного сигналу під
конкретну PCB-геометрію MIKROE-3240 (розташування і топологія котушки, довжина доріжок).
Ці значення специфічні для MIKROE-3240 — не перености для інших плат без калібрування.

⚠️ Потенційний ризик: архітектурний код записує TC1/TC2 але НЕ верифікує запис (LA-5).
   TC1 та TC2 — R/W регістри (SNOSD01D Table 3) → readback можливий.
   Пропуск верифікації малоймовірно призведе до проблем, але є відхиленням від стандарту
   надійності, встановленого для RP_SET.
```

## 3. Фізичні симуляції та моделювання

### SIM-1: Skin depth — виправлення системної помилки SYSTEM_AUDIT

**⚠️ КРИТИЧНА ПОМИЛКА в SYSTEM_AUDIT §3.2:**

SYSTEM_AUDIT заявив δ(Ag, 1 MHz) = 20.4 µm. Правильне значення при 1 MHz = **63.4 µm**.

Верифікація за формулою SNOSD01D (та всіма стандартними довідниками):

```
δ = √(2 / (ω × μ₀ × μr × σ))    де ω = 2πf

Для Ag: f=1 MHz, σ=63×10⁶ S/m, μr=1

ω × μ₀ × σ = 2π×10⁶ × 4π×10⁻⁷ × 63×10⁶
            = 8π² × 10⁻¹ × 63×10⁶
            = 7.896 × 63×10⁶
            = 4.974×10⁸

δ = √(2 / 4.974×10⁸) = √(4.021×10⁻⁹) = 6.342×10⁻⁵ m = 63.4 µm

Перевірка: значення 20.4 µm відповідає δ(Ag) при f = 10 MHz:
  δ(Ag, 10 MHz) = 63.4 / √10 = 20.05 µm ≈ 20.4 µm ✓

→ SYSTEM_AUDIT §3.2 розраховував skin depth при f = 10 MHz,
  але позначив результат як f = 1 MHz.
→ Похибка у 3.16× (= √10), i.e., одна десяткова позиція в частоті.
```

**Значення (LA-2):** Вплив на деякі якісні висновки відсутній (відносні співвідношення між металами зберігаються), але абсолютні числа в SYSTEM_AUDIT §3.2 і похідні посилання на них — неправильні.

### SIM-2: Skin depth при 200–500 kHz (коректні значення для MIKROE-3240)

Оскільки fSENSOR ≈ 200–500 kHz (MIN_FREQ=118 kHz в DIG_CFG + LC-оцінка), перераховуємо при реалістичних частотах.

Масштаб: `δ(f₂) = δ(f₁) × √(f₁/f₂)`, тобто при зниженні частоти δ **збільшується**.

```
Формула: δ[µm] = √(2/(2πf × μ₀ × μr × σ))

Метал         σ [MS/m]  μr    δ @ 1 MHz   δ @ 500 kHz   δ @ 300 kHz   δ @ 200 kHz
──────────────────────────────────────────────────────────────────────────────────────
Ag999         63.0      1.0   63.4 µm      89.6 µm       115.7 µm      141.9 µm
Cu            58.5      1.0   65.8 µm      93.0 µm       120.2 µm      147.4 µm
Au            44.2      1.0   75.7 µm     107.0 µm       138.2 µm      169.4 µm
Al            35.7      1.0   84.2 µm     119.0 µm       153.8 µm      188.5 µm
Ni            14.3     600.0   5.4 µm       7.7 µm          9.9 µm       12.2 µm
Сталь 304      1.37    750.0  15.7 µm      22.2 µm        28.7 µm       35.1 µm
W (вольфрам)  18.9      1.0  115.7 µm     163.6 µm       211.4 µm      259.2 µm

Масштабні коефіцієнти від 1 MHz:  ×1.414    ×1.826    ×2.236
```

**Ключові висновки:**
1. При 300 kHz δ(Ag) = 116 µm — значно більше ніж 37 µm, що SYSTEM_AUDIT виводив масштабуванням від хибного 1 MHz значення.
2. Монета 2 мм товщиною має ~17 skin depths → **повністю "непрозора"** для LDC1101 (65% енергії в першому δ, 87% в перших 2δ).
3. **Вольфрам:** δ = 211 µm при 300 kHz → глибоке проникнення = **підробки з вольфрамовим ядром детектуються** завдяки великій δ порівняно з Au/Ag.
4. Нікель, Сталь: malий δ (9–29 µm) → дуже мала зона взаємодії → слабший магнітний ефект ніж очікувалось на 1 MHz (але μr >> 1 компенсує).

### SIM-3: Провідність сплавів Ag-Cu (критично для нумізматики)

SYSTEM_AUDIT (і попередні аналізи) порівнювали σ чистих металів для оцінки диференціації монет. **Це принципово неправильно** для срібних монет.

```
Ефект Маттісена в твердому розчині Ag-Cu:
  При розчиненні Cu-атомів в Ag-решітці вони виступають центрами розсіювання
  електронів → провідність різко падає навіть при малих концентраціях.

  Лінійне змішування (НЕПРАВИЛЬНО):
    σ_mix ≈ x_Ag × σ_Ag + x_Cu × σ_Cu
    σ(Ag92.5) ≈ 0.925 × 63 + 0.075 × 58.5 ≈ 62.7 MS/m  [НЕВІРНО!]

  Реальна провідність Ag-Cu твердих розчинів (з металовидавних довідників):
    Ag999 (чисте):        σ ≈ 63.0 MS/m  (δ @ 300kHz ≈ 116 µm)
    Ag958 (958 проба):    σ ≈ 40–50 MS/m (δ @ 300kHz ≈ 130–145 µm)
    Ag925 (sterling):     σ ≈ 15–25 MS/m (δ @ 300kHz ≈ 185–240 µm)
    Ag900 (coin silver):  σ ≈ 12–20 MS/m (δ @ 300kHz ≈ 205–265 µm)
    Ag800:                σ ≈  8–14 MS/m (δ @ 300kHz ≈ 250–330 µm)
    Cu (solid):           σ ≈ 58.5 MS/m  (δ @ 300kHz ≈ 120 µm)

  Відношення σ(Ag999)/σ(Ag925) ≈ 3–4×
  → RP(Ag999) значно менший ніж RP(Ag925) [більше втрат = менший RP]

  Відношення σ(Ag925)/σ(Ag900):
    Якщо σ(Ag925)=20, σ(Ag900)=16: різниця ~25% → ΔRP ≈ 12% → реально вимірювано!
    Якщо σ(Ag925)=25, σ(Ag900)=20: різниця ~25% → аналогічно

⚠️ ВИСНОВОК SIM-3 (LA-3): Диференціація Ag925/Ag900 МОЖЕ бути реалістичною
  (не неможливою як можна зрозуміти з SYSTEM_AUDIT), але залежить від:
  1. Реального σ конкретних монет (термічна обробка, деформація-відпал впливають!)
  2. Правильного RESP_TIME (більший = краще SNR, важливо для 12% ΔRP)
  3. Температурного контролю (ΔRP від температури > ΔRP від проби — △T=10°C дає ~4% дрейф)

⚠️ ДОДАТКОВИЙ РИЗИК: Відповідь LDC1101 залежить від СКЛАДУ, а не тільки МЕТАЛУ.
  Срібна монета з відомою пробою ВИМАГАЄ тарованих зразків тієї ж монети,
  не просто довідникових σ цистих елементів.
```

### SIM-5: SNR аналіз — RESP_TIME=6144 vs 768

```
Теорія: RP вимірювання в LDC1101 — це усереднення RESP_TIME циклів осцилятора.
Дискретний шум ~ 1/√N, де N пропорційний RESP_TIME.

SNR(A) / SNR(B) = √(RESP_TIME_A / RESP_TIME_B)

Архітектура (6144) vs MikroE default (768):
  SNR_ratio = √(6144/768) = √8 = 2.83×

  Якщо шум MikroE config: σ_RP = 50 Ohm (assume)
  То в архітектурному config: σ_RP ≈ 50/2.83 ≈ 17.7 Ohm

Значення для Ag925/Ag900 диференціації:
  Якщо ΔRP(Ag925 vs Ag900) ≈ 500–2000 Ohm (при відстані ~1 мм)
  При σ_RP ≈ 18 Ohm: SNR = 27–111 → легко вимірюється
  При σ_RP ≈ 50 Ohm: SNR = 10–40  → вимірюється з усередненням N вимірювань

  N вимірювань за 0.5 с (50 Hz × 0.5 с = 25 reads):
    ΔRP ефективний SNR при N=25: SNR × √25 = SNR × 5
    → навіть при σ_RP=50: ефективний SNR = 50–200 ✅ достатньо

→ ВИСНОВОК SIM-5: Архітектурний вибір RESP_TIME=6144 виправданий для точної
  класифікації монет. MikroE default (768) може не забезпечити потрібного SNR
  для Ag925/Ag900 диференціації.
```

### SIM-6: Температурний дрейф — уточнена модель

```
Вплив температури на RP вимірювання (ΔT від +15°C до +35°C = ΔT=20°C):

1. ЗМІНА σ МЕТАЛУ МОНЕТИ:
   α(Ag) = 3.8×10⁻³ /°C (температурний коефіцієнт опору)
   α(Cu) = 3.9×10⁻³ /°C
   При ΔT = +20°C: Δσ/σ ≈ αΔT = -7.8% (σ падає, RP зростає)

2. ВПЛИВ НА RP:
   RP ∝ 1/σ^0.5 (спрощена модель для thick conductor)
   ΔRP/RP ≈ ½ × Δσ/σ ≈ ½ × 7.8% = ~3.9% зміна RP при ΔT=20°C

3. ПОРІВНЯННЯ З СИГНАЛОМ ДИФЕРЕНЦІАЦІЇ:
   Якщо ΔRP(Ag925 vs Ag900) = 12% (з SIM-3)
   Температурний дрейф за 20°C = ~4%
   → відношення сигнал/дрейф = 12% / 4% = 3× — ГРАНИЧНЕ

   При ΔT=10°C: дрейф ~2%, відношення = 6× — прийнятно з компенсацією
   При ΔT=5°C:  дрейф ~1%, відношення = 12× — добре

4. ВПЛИВ КОТУШКИ (мідний дріт):
   R_dc котушки: +11.7% при ΔT=30°C → Q-фактор падає → fSENSOR дрейфує
   Зміна L_DATA відповідно: значна (~10% кодів при ΔT=30°C)

АРХІТЕКТУРНИЙ ВИСНОВОК:
  Без температурної компенсації:
    → Ag925/Ag900 диференціація ненадійна при ΔT > 5°C між вимірами
    → Ag999/Ag925 диференціація надійна (> 3× різниця σ → > 30–50% ΔRP)
  З температурною компенсацією (відносне вимірювання до baseline):
    → Ag925/Ag900 диференціація можлива при ΔT до ~20°C
    → Потрібен temp sensor (NTC, BME280 або аналогічний)

⚠️ SYS-6 залишається відкритою HIGH знахідкою — підтверджується цим аудитом.
```

### SIM-7: RP_SET=0x26 — аналіз динамічного діапазону

```
RP_SET=0x26: RP_MAX=24 kΩ, RP_MIN=1.5 kΩ
LDC1101 вимірює RP в діапазоні [RP_MIN, RP_MAX].

Оцінка очікуваного RP для різних об'єктів:

  Без об'єкта (повітря): RP = RP_MAX = 24 kΩ (обмеження налаштуванням)
  
  Хороший провідник (Ag999, Au, Cu) вплотну до котушки:
    → Значні eddy current → великі втрати → RP значно ПАДАЄ
    → Типово: RP ≈ 500 – 5 000 Ω [в межах [1500, 24000] → можливо на межі!]
   
  Ag925 (нижча σ, більший RP):
    → RP ≈ 1 000 – 8 000 Ω

  Ag900 (ще нижча σ):
    → RP ≈ 1 200 – 10 000 Ω

  Чиста мідь (висока σ, RP може бути < RP_MIN):
    → RP < 1 500 Ω? → STATUS_NO_OSC ризик!

  Феромагнітне залізо (мала σ, великий μr):
    → Інший механізм домінує (μr), RP може бути в середині діапазону

⚠️ РИЗИК LA-MEDIUM: При вимірюванні монет з дуже висококою σ (Cu, Ag999 вплотну)
  RP може впасти нижче RP_MIN=1.5 kΩ → сенсор входить в зону "NO_OSC" або
  дає насичені показання. RP_SET може потребувати налаштування.
  
  ADR-LDC-001 передбачає регулювання через `rp_set` в конфігурації —
  правильний підхід. Але алгоритм recovery при NO_OSC недостатній (LA-8).

РЕКОМЕНДАЦІЯ: При першому включенні сенсора з реальними монетами:
  1. Зареструвати RP raw для кожного металу без рухомих частин
  2. Если RP < 2 kΩ або STATUS_NO_OSC → підвищити RP_MAX (зменшити rp_set)
  3. Якщо RP ≈ RP_MAX=24 kΩ для всіх монет → знизити RP_MAX
```

### SIM-8: LHR mode (24-bit L) — потенціал для диференціації

```
LHR (High-Resolution L mode) — 24-bit вимірювання індуктивності.
Стандартний режим: L_DATA = 16 bit (65535 рівнів).
LHR: 24 bit = 16.77M рівнів → 256× вища роздільна здатність.

Але: L_DATA вимірює fSENSOR, яка залежить від L_eff(монета).
  ΔL через μr монети: тільки для феромагнітних металів!
  
Для Ag, Cu, Au: μr ≈ 1 → ΔL = мала (лише через геометрію, не μr)
Для Fe, Ni: μr >> 1 → ΔL = велика (надійна класифікація без LHR)

ВИСНОВОК SIM-8: LHR mode корисний для:
  ✓ Точного вимірювання fSENSOR без монети → обчислити C_SENSOR, L_coil
  ✓ Тонкої феромагнітної класифікації (відрізнити Ni від Fe, якщо потрібно)
  ✗ НЕ допомагає диференціювати Ag925/Ag900 (обидва: μr ≈ 1, ΔL мала)
  ✗ НЕ підвищує SNR для RP-каналу (тільки L-канал)

РЕКОМЕНДАЦІЯ: Реалізувати LHR mode для:
  - Калібрування на початку роботи (точна fSENSOR без монети)
  - Перевірки "чи монета феромагнітна" (L-код різко зростає при Fe/Ni)
  - Це доповнює, а не замінює RP-канал
```

### SIM-9: Деградація сигналу з відстанню — вплив for монет

```
Модель RP(d): ΔRP(d) ≈ ΔRP(0) × (1 + d/d_char)^(-n)   де n=2..3

При правильних skin depth (300 kHz):
  d_char ≈ δ = 116 µm для Ag при 300 kHz (корелює з depth of interaction)

  d = 0 мм:   ΔRP = ΔRP(0) = 100% (базова)
  d = 0.5 мм: k ≈ 0.63–0.75  (25–37% втрата)
  d = 1 мм:   k ≈ 0.40–0.52
  d = 3 мм:   k ≈ 0.10–0.16

SPACER TOLERANCE при d=1 мм:
  Похибка ±0.05 мм: Δk ≈ dk/dd × 0.05
  ≈ 2–3% зміна від номінального RP

  Якщо ΔRP(Ag925 vs Ag900) = 12%:
  Spacer похибка 0.05 мм додає ~2–3% невизначеності → коефіцієнт перекриття ~0.25
  → SYS-13 підтверджується: spacer tolerance ±0.05 мм є фундаментальним обмеженням

⚠️ Наслідок для DIY spacers: 3D-друкована деталь точністю ±0.1–0.2 мм
   дає к ≈ 5–8% похибки → маскує Ag925/Ag900 різницю.
   Тільки механічно точні (фрезеровані) spacers підходять для проб.
```

### SIM-10: Виявлення вольфрамових підробок

```
Вольфрам (W, σ=18.9 MS/m, μr=1.0) — матеріал підробок під Au/Ag
  через близькість густини: ρ(W)=19.3 г/см³ vs ρ(Au)=19.3 г/см³

При f = 300 kHz:
  δ(W, 300 kHz)  = 211 µm  (великий!)
  δ(Au, 300 kHz) = 138 µm
  δ(Ag, 300 kHz) = 116 µm

  σ(W) = 18.9 MS/m << σ(Au) = 44.2 MS/m

  RP(W) >> RP(Au) (вольфрам погано провідний → менші eddy currents → великий RP)

Оцінка ΔRP(Au vs W):  
  RP ∝ 1/σ^0.5 (спрощена thick-conductor модель)
  RP(W)/RP(Au) ≈ √(σ(Au)/σ(W)) = √(44.2/18.9) = √2.34 = 1.53×
  → W має RP вищий за Au на ~53% → ЛЕГКО розрізниється ✅

  З чистого Au: RP менший через більшу σ.
  W-ядро покрите Au плακою: сигнал залежить від товщини плакування та δ

ВИСНОВОК SIM-10: Виявлення W-підробок за RP-каналом є фізично обґрунтованим.
  Потрібне тільки калібрування на еталонних монетах.
  ✅ Цей use case НЕ вимагає Ag925/Ag900 точності — великий ΔRP достатньо.
```

### SIM-11: SPI timing бюджет

```
Перевірка відповідності PLUGIN_CONTRACT.md §1.2:

read() total:
```

### SIM-12: convTimeMs() — аналіз worst-case

```cpp
uint32_t convTimeMs() const {
    static const uint32_t cycles[] = {192, 192, 192, 384, 768, 1536, 3072, 6144};
    uint8_t idx = respTimeBits & 0x07;
    return (cycles[idx] + 499) / 500 + 2;  // "Worst case: fSENSOR = 500 kHz"
}
```

```
Аналіз: функція моделює worst-case як fSENSOR=500 kHz → 
  час = ceiling(cycles/500kHz) + 2ms margin

Реальний worst case: fSENSOR = 200 kHz (MIN_FREQ=118 kHz → може бути нижче 500 kHz)

При RESP_TIME=6144, fSENSOR=200 kHz:
  Реальний час:    6144 / (3 × 200000) = 10.24 мс
  Функція повертає: (6144 + 499) / 500 + 2 = 13 + 2 = 15 мс

  15 мс > 10.24 мс → функція ЗАВИЩУЄ час → wait в calibrate() довший ніж треба
  → БЕЗПЕЧНО (дані завжди готові), але коментар "500 kHz worst case" НЕПРАВИЛЬНИЙ

При RESP_TIME=6144, fSENSOR=300 kHz:
  Реальний час:     6144 / (3 × 300000) = 6.83 мс
  Функція повертає: 15 мс → чекаємо 15 мс замість 6.83 мс → 2.2× надлишок

→ ЗНАХІДКА LA-9 (LOW): коментар та логіка worst-case невірні. Функція безпечна
  але вводить в oману. Правильний worst case: 200 kHz.
  
  Виправлення коментаря: "// Worst case: fSENSOR = 200 kHz (MIN_FREQ=118 kHz)"
  Зміна логіки для точності: (cycles[idx] + 199) / 200 + 2
  
  Але: зміна збільшить час очікування в calibrate()/runSelfTest(). Поточна версія
  є safe (консервативно завищує). Виправляти необов'язково.
```

## 4. Аналіз реалізації: LDC1101Plugin

### 4.1 LA-1 [CRITICAL]: DIG_CONFIG verify — регресійний баг хвилі 3

Хвиля 3 додала `MIN_FREQ=0xD0` в DIG_CFG, але не оновила логіку верифікації:

```cpp
// Після хвилі 3:
uint8_t digCfg = 0xD0 | (respTimeBits & 0x07);   // digCfg ∈ [0xD0..0xD7]
spiWrite(REG_DIG_CONFIG, digCfg);

uint8_t digReadBack = spiRead(REG_DIG_CONFIG);
if ((digReadBack & 0x07) != digCfg) {              // ← БАГ!
    return false;  // завжди виконується!
}
```

**Аналіз:**
```
digCfg     = 0xD0..0xD7  (8-бітне значення з MIN_FREQ в старших бітах)
(digReadBack & 0x07) = 0x00..0x07  (тільки молодші 3 біти RESP_TIME)

Умова: maske_3bit != 8bit_value
  0x07 != 0xD7 → TRUE → помилка завжди спрацьовує!

Результат: configureSensor() ЗАВЖДИ повертає false.
  → initialize() ЗАВЖДИ повертає false.
  → LDC1101Plugin НІКОЛИ не ініціалізується.
```

**Виправлення:**
```cpp
// Правильно — порівнювати весь байт:
if (digReadBack != digCfg) {
```

**або** (якщо навмисно перевіряти тільки RESP_TIME bits):
```cpp
if ((digReadBack & 0x07) != (digCfg & 0x07)) {
```

**Рекомендація:** Виправити перед хвилею 4. Одна строка, не потребує тестування на пристрої.

### 4.2 LA-5 [MEDIUM]: TC1/TC2 — відсутня верифікація запису

```cpp
// Поточний код (after wave 3):
spiWrite(REG_RP_SET, rpSetValue);    // + verify readback ✅
spiWrite(REG_TC1, 0x1F);             // ← НЕ верифікується ⚠️
spiWrite(REG_TC2, 0x3F);             // ← НЕ верифікується ⚠️
```

TC1/TC2 — конфігураційні R/W регістри (підтверджено SNOSD01D Table 3). Readback можливий.

**Ризик:** Мінімальний при справному SPI. Але наявна верифікація RP_SET та DIG_CFG при відсутній для TC1/TC2 є логічною непослідовністю. При SPI-проблемах (loose connection, noise on bus) TC1/TC2 можуть бути записані невірно без діагностики.

**Рекомендація:** Додати verify аналогічно RP_SET:
```cpp
spiWrite(REG_TC1, 0x1F);
if (spiRead(REG_TC1) != 0x1F) { ...error... }
spiWrite(REG_TC2, 0x3F);
if (spiRead(REG_TC2) != 0x3F) { ...error... }
```

### 4.3 LA-6 [MEDIUM]: checkCalibration() — надто широкі межі

```cpp
bool checkCalibration() override {
    return calibrationRpBaseline > 100.0f && calibrationRpBaseline < 60000.0f;
}
```

Умова `(100, 60000)` покриває практично весь 16-bit RP діапазон (max 65535). Це означає, що `checkCalibration()` повертає `true` для будь-якої "успішної" калібрації, навіть якщо RP_baseline поза очікуваним діапазоном для нумізматичних монет.

**Рекомендація:** Після емпіричного вимірювання діапазону RP для MIKROE-3240, звузити межі до реалістичних значень (наприклад, `2000 < baseline < 30000` якщо виміряний діапазон підтверджує це).

### 4.4 LA-7 [MEDIUM]: update() відсутній DRDYB timeout

```cpp
if (status & STATUS_DRDYB) {
    return;  // Конверсія ще триває — норма, чекаємо наступного update()
}
```

При pSENSOR=200 kHz, RESP_TIME=6144: час конверсії = 10.24 мс. Якщо `update()` викликається @ 50 Hz (20 мс), то DRDYB завжди буде 0 при кожному виклику — без проблем.

**Але:** якщо `update()` викликається частіше (100+ Hz або в тесті), DRDYB може бути 1 при кожному виклику. Поточний код просто повертається — немає лічильника "скільки разів consecutively DRDYB=1", після якого слід логувати попередження або повернути DEGRADED.

**Рекомендація:** Додати простий лічильник stale_count → якщо > 10 consecutive "DRDYB=1" → log warning. Вказує на неправильний fSENSOR або frozen conversion.

### 4.5 LA-8 [LOW]: NO_OSC recovery — відсутня диференційна діагностика

```cpp
if (status & STATUS_NO_OSC) {
    diag.failedReads++;
    diag.status    = HealthStatus::NOT_FOUND;
    diag.lastError = {6, "Coil not oscillating — check wiring or RP_SET"};
    ctx->log->warning(getName(), diag.lastError.message);
    return;
}
```

`STATUS_NO_OSC` означає або:
1. RP_SET знаходиться поза RP котушки (неправильна конфігурація)
2. Котушка не підключена або пошкоджена
3. Interference/noise на живленні (transient)

Поточний код не розрізняє ці випадки. Після N*(NO_OSC) рекомендується:
- Спробувати альтернативний RP_SET (авто-підбір)
- Або встановити HealthStatus::SENSOR_FAULT після >100 no-osc

**R6 (з LA-3 + SIM-6):** До збору fingerprint data — виміряти:
- fSENSOR = fCLKIN × RESP_TIME / (3 × L_DATA_empty) — 1 вимірювання LHR mode
- C_SENSOR через зворотний розрахунок  
- Температурний дрейф baseline за 20°C діапазон

**R7 (з LA-6):** Звузити checkCalibration() межі після отримання R5 даних.

### Довгострокові (> P-3)

**R8 (з SIM-6):** Температурна компенсація. Мінімум: зберігати temp_c у fingerprint record і відхиляти порівняння при |ΔT| > 5°C. Максимум: реальна NTC компенсація σ(T).

**R9 (з LA-8):** LHR mode для калібрування fSENSOR на початку кожної сесії. Потребує ~10 мс, виконувати в `initialize()` перед Active mode.

## 8. Загальний висновок

### Що добре

Архітектура LDC1101Plugin є **найзрілішим прикладом embedded plugin-системи** в проекті. State Machine стратегія (Strategy A) — правильна для нумізматичного use case. Burst read з правильним LSB→MSB порядком та CS low протягом усієї транзакції — технічно бездоганна реалізація.

Після хвилі 3: синхронізація з MikroE SDK (TC1/TC2/RP_SET/MIN_FREQ) усунула основні конфігураційні ризики.

### Що потребує виправлення

**Один критичний баг** (LA-1, одна строка коду) повністю блокував ініціалізацію сенсора. **✅ Виправлено в хвилі 3 (commit d73427c).**

**Фізичний аналіз** у SYSTEM_AUDIT §3.2 мав системну помилку в skin depth (10 MHz замість 1 MHz). **✅ Виправлено в хвилі 4.** Якісні висновки про класифікаційний простір в цілому правильні.

**Diференціація Ag925/Ag900** фізично можлива (alloy scattering дає ~25% ΔRP), але вимагає:
- Точного temperature control або компенсації
- Mechanically precise spacers (≤ ±0.05 мм)
- Емпіричної верифікації на реальних монетах

### Готовність до імплементації

**Після виправлень хвиль 3+4 (`configureSensor()` fix + ADR-RESP-001 + TC1/TC2 verify + staleCount):**
