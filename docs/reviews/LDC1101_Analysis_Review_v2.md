## 3. Що зроблено відмінно

### ✅ Верифікація datasheet — еталонний стандарт

Документ містить точні посилання на конкретні розділи TI LDC1101 Datasheet (SNOSD01D): розділ 8.5.1, 8.6.5, 8.6.19, 8.6.21, Table 3, Table 20. Жодне твердження не є припущенням. Порівняльні таблиці v1.0 vs datasheet у секції 5 — чудовий приклад технічної рецензії.

### ✅ Виявлення 7 критичних помилок v1.0

Секція 5 вичерпно і з поясненнями документує всі помилки попередньої версії. Особливо цінно:
- Причинно-наслідковий зв'язок (не просто "неправильно", а чому і який наслідок)
- Порівняльні блоки коду "v1.0 НЕПРАВИЛЬНО" vs "Правильно"
- Зведена таблиця з критичністю кожної проблеми

### ✅ Секція 2: Фізика і часові характеристики

Детальний розбір формули `ConversionTime = RESP_TIME_cycles / (3 × fSENSOR)` з верифікованими числами. Порівняльна таблиця котушок 2 MHz vs 500 kHz дає розробнику практичне розуміння trade-off без необхідності читати datasheet.

### ✅ SPI Burst читання — критична деталь пояснена

Чітке пояснення чому LSB читається першим з цитатою datasheet і практичним прикладом burst транзакції. Без цього пояснення розробник майже напевно прочитав би MSB першим.

### ✅ Секція 6: Дві стратегії — правильний вибір

Порівняння Стратегії A (State Machine, 50 Hz) і Стратегії B (FreeRTOS task, 1000 Hz) з часовими діаграмами. Рекомендація Стратегії A для v1 з чітким обґрунтуванням ("50 вимірювань за 0.5 с достатньо") — правильне архітектурне рішення.

### ✅ Секція 7: SPI mutex — превентивний підхід

Правильно обґрунтовано чому `spiMutex` потрібен превентивно навіть коли Стратегія A сама по собі race-free. Аргумент "перший розробник з async task отримає race condition без попередження" — промислово зрілий підхід до архітектурних рішень.

## 4. Критичні проблеми

### 🔴 Критична 1: `portMAX_DELAY` порушує `PLUGIN_CONTRACT.md`

**Знаходиться в:** Секція 9, методи `update()` і `read()`

```cpp
// update() — рядок з оновленням кешу:
xSemaphoreTake(dataMutex, portMAX_DELAY);   // ❌ ПОРУШЕННЯ КОНТРАКТУ!
cached.rpRaw = rpRaw;
// ...
xSemaphoreGive(dataMutex);

// read() — рядок з читанням кешу:
xSemaphoreTake(dataMutex, portMAX_DELAY);   // ❌ ПОРУШЕННЯ КОНТРАКТУ!
```

**Чому критично:**  
`PLUGIN_CONTRACT.md` секція 1.3 явно забороняє `portMAX_DELAY`:
- `update()` max **10 ms** — якщо інший task тримає `dataMutex`, `update()` заблокується назавжди
- `read()` max **5 ms** — аналогічно

Це ж правило встановлено в `LOGGER_ARCHITECTURE.md` і `CREATING_PLUGINS.md`. Документ описує ці правила в секції 7 і 8, але код в секції 9 їх порушує.

**Виправлення:**
```cpp
// update():
if (xSemaphoreTake(dataMutex, pdMS_TO_TICKS(5)) != pdTRUE) {
    diag.failedReads++;  // mutex timeout — drop update, не блокуємо
    return;
}
cached.rpRaw = rpRaw;
// ...
xSemaphoreGive(dataMutex);

// read():
if (xSemaphoreTake(dataMutex, pdMS_TO_TICKS(3)) != pdTRUE) {
    return {0, 0, 0.0f, millis(), false};  // timeout — повернути invalid
}
```

### 🔴 Критична 2: `reconfigureSensor()` — undefined method, compile error

**Знаходиться в:** Секція 10, демонстрація fast/precise режимів

```cpp
void setFastMode() {
    respTimeBits = 0x02;
    reconfigureSensor();   // ❌ НЕ ВИЗНАЧЕНО в секції 9!
}

void setPreciseMode() {
    respTimeBits = 0x07;
    reconfigureSensor();   // ❌ НЕ ВИЗНАЧЕНО в секції 9!
}
```

**Чому критично:**  
Секція 10 містить код який не є автономним прикладом — він є частиною класу `LDC1101Plugin` (той самий стиль, посилається на `respTimeBits`). Розробник, який скопіює цей код, отримає помилку компіляції. `reconfigureSensor()` є `spiWrite(REG_START_CONFIG, FUNC_SLEEP) → configureSensor() → spiWrite(REG_START_CONFIG, FUNC_ACTIVE)` — це неважко здогадатись, але документ не повинен залишати розробника здогадуватись.

**Виправлення:**  
Або додати реалізацію `reconfigureSensor()` в секцію 9 (приватний метод), або змінити секцію 10 на псевдокод із позначкою "реалізація через `configureSensor()` з секції 9".

## 5. Середні проблеми

### 🟡 Середня 1: `spiRead()` і `spiWrite()` — відсутні guards

**Знаходиться в:** Секція 9, нижні private методи

```cpp
uint8_t spiRead(uint8_t reg) {
    uint8_t result;
    ctx->spi->beginTransaction(...);  // ❌ якщо ctx->spi == nullptr → crash
    // ...
}
```

`readMeasurementBurst()` має guard:
```cpp
if (!ctx || !ctx->spi || csPin < 0) return false;  // ✅ є
```

Але `spiRead()` і `spiWrite()` — ні. При цьому `spiRead()` викликається в `initialize()` ще до повної валідації (перевірка POR): якщо `PluginContext` передати з `spi == nullptr`, отримаємо null dereference на першому виклику SPI.

**Виправлення:**
```cpp
uint8_t spiRead(uint8_t reg) {
    if (!ctx || !ctx->spi || csPin < 0) return 0xFF;  // guard
    // ...
}

void spiWrite(uint8_t reg, uint8_t value) {
    if (!ctx || !ctx->spi || csPin < 0) return;  // guard
    // ...
}
```

---

### 🟡 Середня 2: Ігнорування `return value` `readMeasurementBurst()` в `runSelfTest()`

**Знаходиться в:** Секція 9, метод `runSelfTest()`

```cpp
for (int i = 0; i < 5; i++) {
    delay(convTimeMs() + 2);
    uint16_t rp, l;
    readMeasurementBurst(rp, l);   // ❌ return value ігнорується
    readings[i] = rp;              // rp містить сміття якщо SPI провалився
}
```

Порівняно з `calibrate()` де все правильно:
```cpp
if (readMeasurementBurst(rp, l) && rp > 0 && rp < 65535) {  // ✅
```

Якщо SPI не відповідає під час `runSelfTest()`, `rp` матиме значення з попередньої ітерації або невалідний garbage. Стабільність вважатиметься нормальною — хибнопозитивний self-test.

**Виправлення:**
```cpp
for (int i = 0; i < 5; i++) {
    delay(convTimeMs() + 2);
    uint16_t rp, l;
    if (!readMeasurementBurst(rp, l)) {
        ctx->log->error(getName(), "  SPI read failed during stability test");
        return false;
    }
    readings[i] = rp;
}
```

### 🟡 Середня 3: Назви констант різняться між секцією 4 і секцією 9

**Секція 4 (повна карта регістрів):**
```cpp
static const uint8_t FUNC_MODE_ACTIVE    = 0x00;
static const uint8_t FUNC_MODE_SLEEP     = 0x01;
static const uint8_t FUNC_MODE_SHUTDOWN  = 0x02;
static const uint8_t STATUS_POR_READ     = 0x01;
```

**Секція 9 (реалізація класу):**
```cpp
static const uint8_t FUNC_ACTIVE         = 0x00;   // ≠ FUNC_MODE_ACTIVE
static const uint8_t FUNC_SLEEP          = 0x01;   // ≠ FUNC_MODE_SLEEP
static const uint8_t STATUS_POR          = 0x01;   // ≠ STATUS_POR_READ
```

Розробник, котрий слідує документу між секціями, побачить різні назви для тих самих констант. Це не баг в рантаймі, але ускладнює осмислення документа і підвищить ризик помилок при копіюванні.

**Виправлення:** Консолідувати назви — або привести секцію 4 до скорочених форм, або секцію 9 до повних. Рекомендую нові повні форми з секції 4 як більш читабельні (`FUNC_MODE_SLEEP` краще за `FUNC_SLEEP`).

### 🟡 Середня 4: `RP_SET` — критичний регістр без конфігурації

**Знаходиться в:** Секція 9, `configureSensor()`

`REG_RP_SET = 0x01` — регістр що визначає динамічний діапазон паралельного опору котушки. Якщо він не налаштований правильно, `STATUS_NO_OSC` спрацює і сенсор не видасть жодного вимірювання.

`configureSensor()` записує `DIG_CONFIG`, `D_CONFIG`, `ALT_CONFIG` — але `RP_SET` **не торкається**, залишаючи default значення при power-on. З datasheet (Table 4):
> *"RP_SET: Configures the RP measurement dynamic range driven by RPMIN and RP_SET codes"*

Default значення при POR — **0x07** (RPMIN = 0, RP_SET = 7, що відповідає найширшому діапазону). Для більшості котушок це може працювати, але:
1. Документ взагалі не згадує цей регістр у процедурі конфігурації
2. У `data/plugins/ldc1101.json` (розділ 8) не передбачено параметра для RP_SET
3. Якщо котушка не вкладається в default діапазон — розробнику буде важко знайти причину `NO_SENSOR_OSC`

**Рекомендація:** Додати `rp_set` в `ldc1101.json` з default значенням 7 і коротким коментарем що і коли треба змінювати.

## 6. Незначні проблеми

### 🔵 Незначна 1: LHR-регістри документовані але роль незрозуміла

Секція 4 документує 9 регістрів LHR mode (0x30–0x3B) ohne жодних пояснень. Ніде в документі (секції 6, 9, 10) не згадується: LHR не планується для v1? Або це просто повна карта для довідки? Розробник може витратити час на вивчення LHR thinking воно потрібне.

**Рекомендація:** Додати одне речення в кінці секції 4: *"LHR (High-Resolution L mode) — опціональний режим для точних вимірювань індуктивності. Не використовується в v1.0 реалізації, документовано для майбутнього використання."*

### 🔵 Незначна 2: Footer посилається на неіснуючу назву файлу

```
Попередня версія: LDC1101_Architecture_Analysis_v1.0.0.md
```

Фактична назва файлу в репозиторії: `LDC1101_Architecture_Analysis.md` (без версії в назві, зараз в `obsolete/`).

### 🔵 Незначна 3: `convTimeMs()` — неочевидна ініціалізація масиву для Reserved індексів

```cpp
static const uint32_t cycles[] = {192, 192, 192, 384, 768, 1536, 3072, 6144};
//                                 ^ ^ ^— індекси 0 і 1 = Reserved в datasheet
```

Значення 192 для індексів 0 і 1 є graceful handling для "Reserved" RESP_TIME значень, але це не задокументовано. Розробник може порахувати що cycles[0] і cycles[1] мають різні значення.

**Рекомендація:** Додати `// idx 0,1: Reserved в datasheet → використовуємо мінімум cycles як safe fallback`.

### 🔵 Незначна 4: INTB pin — відсутнє пояснення чому не використовується

`REG_INTB_MODE = 0x0A` визначено, але INTB interrupt pin не використовується — замість нього polling STATUS у кожному `update()`. Для Стратегії A це правильне рішення, але коментар в коді відсутній. Майбутній розробник що побачить `REG_INTB_MODE` в списку констант може запитатись: "навіщо він там якщо не використовується?"

## 7. Структурні зауваження

### ⚪ Зауваження 1: Секція з проблемами v1.0 могла бути в окремому розділі "Changelog"

Секція 5 ("Проблеми поточної реалізації") займає значну частину документа і описує помилки попередньої версії. Це корисно для розуміння контексту, але якщо документ буде використовуватись як reference —  розробник буде читати опис того, чого вже немає. Для нового читача (який не знайомий з v1.0) секція 1 (критичне виправлення) і секція 5 разом створюють враження що документ про проблеми, а не про рішення.

**Рекомендація:** Немає потреби перейменовувати секцію — це стиль вже встановлений в проекті. Але можна додати в початок секції 5 рядок: *"Усі проблеми нижче виправлені в v2.0.0 — секція 9 містить коректну реалізацію."*

### ⚪ Зауваження 2: "Рекомендована реалізація" в секції 9 — без зв'язку з PLUGIN_CONTRACT обмеженнями

Секція 9 є еталонним кодом для розробника. Але вона не містить жодного посилання на `PLUGIN_CONTRACT.md` з нагадуванням про обмеження що діють на весь цей код (max 10ms, no Wire.begin(), memory limit). Оскільки код сам по собі порушує `portMAX_DELAY`, посилання тут виправдане.

**Рекомендація:** Додати в початок секції 9 блок:
```
⚠️ Ця реалізація підкоряється PLUGIN_CONTRACT.md:
- update() max 10 ms
- read() max 5 ms  
- portMAX_DELAY ЗАБОРОНЕНО (5ms timeout з drop)
- No Wire.begin()/SPI.begin() (вже ініціалізовано системою)
```
